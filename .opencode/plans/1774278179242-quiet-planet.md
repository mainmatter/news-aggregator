# Plan: simplify edition article ordering

## Recommended approach

- Change `daily_edition_article.position` from an integer to a floating-point numeric column in SQLite so midpoint-based reordering has much more room before a rebalance is needed.
- Use sparse floating-point positions with a shared step constant (recommended: `1024`).
- Treat the stored `position` as opaque ordering metadata, not as the visible rank shown in the UI.
- In the common reorder case, update only the moved row:
  - move up one slot -> new `position = (position_of_item_above + position_of_item_two_above_or_start_gap) / 2`
  - move down one slot -> new `position = (position_of_item_below + position_of_item_two_below_or_end_gap) / 2`
  - move to the start/end -> use `neighbor.position +/- POSITION_STEP`
- If two neighboring positions become too close for safe midpoint insertion, rebalance the edition once, then recompute the midpoint and finish the move.

## Important edge cases

- first/last item moves
- editions with 0 or 1 article
- repeated moves that eventually exhaust the available midpoint gap
- floating-point precision getting too tight between adjacent rows
- unique `(daily_edition_id, position)` collisions during reorder
- optimistic UI staying in sync after the server refresh returns sparse positions
