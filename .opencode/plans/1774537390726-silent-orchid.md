# Generate Daily Edition Workflow Plan

## Recommended approach

Use Workflow Dev Kit as the durable orchestrator and keep the workflow itself under `src/workflows/generate_daily_edition`. The app only needs an authenticated manual trigger for `user_id + edition_date`; the workflow can later be reused by a scheduled trigger without changing its core logic.

For each active user source, the workflow will create a Workflow Dev Kit webhook with `createWebhook()`, launch a Vercel sandbox for that source, and pass the sandbox the webhook URL plus a derived HMAC secret. The sandbox will use the Opencode SDK to visit the source page, let AI extract the relevant links, visit those links, build structured summaries, and POST the result directly to the generated webhook URL. The workflow will `await` those webhook requests directly and only persist once the source results have been collected.

Reuse the existing schema only: `article`, `source_article`, `daily_edition`, and `daily_edition_article`. Support reruns with an explicit `replace_existing` flag so we can rebuild the edition for the same date without adding a workflow-run table.

## File responsibilities

- `src/workflows/generate_daily_edition/index.ts`
  - export the workflow entrypoint
  - orchestrate source loading, webhook creation, sandbox launch, webhook awaiting, and final persistence
- `src/workflows/generate_daily_edition/steps/get_user_sources.ts`
  - load the current user's active sources from `user_source` joined with `source`
- `src/workflows/generate_daily_edition/steps/launch_source_sandbox.ts`
  - create a Vercel sandbox for one source
  - derive the per-webhook HMAC secret from `WEBHOOK_SECRET` and the generated webhook token
  - inject the webhook URL, derived secret, and Opencode env into the sandbox
  - start the sandbox task that performs extraction/summarization
- `src/workflows/generate_daily_edition/steps/consume_source_webhook.ts`
  - accept the `Request` returned by the workflow webhook
  - verify the sandbox signature and timestamp using the webhook token and `WEBHOOK_SECRET`
  - parse and validate the callback payload
- `src/workflows/generate_daily_edition/sandbox/runner.ts`
  - this is the code executed inside the Vercel sandbox
  - fetch the source page
  - use the Opencode SDK to extract relevant links from the source page
  - visit the discovered links and use Opencode again to produce structured summaries
  - POST the final payload to the workflow webhook URL with the HMAC signature headers
- `src/workflows/generate_daily_edition/steps/persist_edition.ts`
  - dedupe and canonicalize URLs across all source results
  - upsert `article` rows and `source_article` links
  - create or replace the target `daily_edition`
  - write ordered `daily_edition_article` rows and set `generated_at`
- `src/lib/server/edition_generation.ts`
  - provide a reusable server helper to start the workflow now and from future scheduling code

## Data flow

1. Add a manual authenticated trigger for a specific `edition_date`.
2. The trigger starts `generate_daily_edition_workflow` with the current user's id and the selected date.
3. The workflow loads all active sources for that user.
4. For each source, the workflow creates a webhook with `createWebhook()`, then launches a sandbox with that webhook URL and a derived signing secret.
5. The sandbox visits the source page, lets Opencode extract relevant links, visits each link, and builds structured article summaries.
6. The sandbox POSTs the result directly to the generated webhook URL.
7. The workflow receives the request by awaiting the webhook, verifies the signature in a step, and collects the typed source result.
8. After all source runs have completed or timed out, the workflow persists the final edition into the existing tables.

## Security and env strategy

- Keep `WEBHOOK_SECRET` as the root secret in app env only.
- After `createWebhook()` returns a token, derive a scoped secret such as `HMAC_SHA256(WEBHOOK_SECRET, webhook.token)`.
- Pass only that derived secret into the sandbox.
- The sandbox signs `timestamp + "." + raw_body` with the derived secret and sends headers such as:
  - `x-news-callback-timestamp`
  - `x-news-callback-signature`
- When the workflow receives the webhook request, `consume_source_webhook.ts` recomputes the derived secret from `WEBHOOK_SECRET` and `webhook.token`, rejects stale timestamps, verifies the signature with a timing-safe comparison, and only then accepts the payload.
- The generated webhook URL from `createWebhook()` is used directly; no custom callback route is needed, and the sandbox can post to it in fire-and-forget mode.
- Document the required env in `.env.example`, including:
  - `WEBHOOK_SECRET`
  - the Opencode API/env values needed in the sandbox
  - the Vercel sandbox credentials needed server-side

## Persistence rules

- Reuse `daily_edition` as the canonical stored output of the run.
- Reuse `article.summary`, `article.category`, `article.published_at`, and `article.summarized_at` for AI-enriched article data.
- Reuse `source_article` to record which source surfaced which article.
- On rerun for the same `user_id + edition_date`, either fail fast if an edition already exists or allow replacement only when `replace_existing` is true.
- If replacing, delete existing `daily_edition_article` rows in a transaction and rebuild them from the workflow result.

## Main edge cases to handle during implementation

- arbitrary source pages can be noisy or JS-heavy, so the Opencode prompt needs to strongly constrain which links count as relevant news/article links
- duplicate article URLs can appear across multiple sources, so canonicalization must happen before inserts
- webhook retries or duplicate posts must be idempotent at the persistence layer
- reruns should not overwrite curated edition content unless `replace_existing` is explicit
- some sources will fail or return no useful links, so the workflow should tolerate partial results and still complete deterministically
