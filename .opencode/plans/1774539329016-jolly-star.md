# Generate Daily Edition Workflow Plan

## Recommended approach

Add a Workflow Dev Kit orchestration flow under `src/workflows/generate_daily_edition` and start it from an authenticated remote form. In v1, the user-facing trigger lives on `src/routes/news/[[date=date]]/+page.svelte` when the selected edition is missing or empty. The workflow creates or updates `daily_edition` immediately with `status = 'generating'`, launches one sandbox run per active source, waits for signed webhook callbacks, and persists the final lineup only in the last transaction.

Keep `replace_existing` in the server and workflow input types for future use, but do not expose rerun UI in v1.

## Product decisions locked in

- Successful generation ends with `daily_edition.status = 'published'`.
- If at least one source succeeds, publish a partial edition from the successful source results.
- If zero sources succeed and an edition already existed, leave the current edition content unchanged.
- If zero sources succeed and no edition existed yet, create a new empty published edition with explicit copy explaining there were no new articles.
- Freshness should target the requested edition date with a 48-hour lookback.
- When a canonical URL already exists in `article`, reuse existing enrichment when the stored row is complete enough instead of re-summarizing every run.
- Status remains edition-level only in v1; no per-source live progress UI.

## File responsibilities

- `src/lib/editions.remote.ts`
  - add an authenticated remote form such as `start_daily_edition_generation`
  - validate `edition_date` and optional `replace_existing`
  - only expose the first-run path in v1 UI, but keep `replace_existing` in the server contract
  - delegate workflow startup to `src/lib/server/edition_generation.ts`
- `src/lib/server/edition_generation.ts`
  - provide a reusable helper that starts the workflow with `user_id`, `edition_date`, and `replace_existing`
  - enforce guardrails such as "do not start another run for the same edition while status is `generating`"
- `src/workflows/generate_daily_edition/index.ts`
  - export the workflow entrypoint
  - keep only orchestration in the workflow body
  - for each source, create a workflow webhook, call a step that launches the sandbox, await the webhook request, and pass the request into a verification/parsing step
- `src/workflows/generate_daily_edition/steps/prepare_generation.ts`
  - load the existing edition state for the requested `user_id + edition_date`
  - create or update the `daily_edition` row immediately with `status = 'generating'`
  - preserve enough previous edition metadata to restore unchanged state on zero-success reruns
- `src/workflows/generate_daily_edition/steps/get_user_sources.ts`
  - load active `user_source` rows joined with `source`
  - return the canonical source URLs and display metadata needed by the workflow
- `src/workflows/generate_daily_edition/steps/launch_source_sandbox.ts`
  - create one Vercel sandbox for one source
  - derive a per-webhook HMAC secret from `WEBHOOK_SECRET` and the generated webhook token
  - inject the webhook URL, derived secret, edition date window, and Opencode env into the sandbox
  - execute `sandbox/runner.ts`
- `src/workflows/generate_daily_edition/steps/consume_source_webhook.ts`
  - accept the `Request` returned by the awaited workflow webhook
  - verify timestamp freshness and HMAC signature using the webhook token plus `WEBHOOK_SECRET`
  - parse and validate the callback payload into a typed source result
- `src/workflows/generate_daily_edition/steps/persist_edition.ts`
  - normalize and dedupe canonical URLs across all successful source results
  - reuse existing `article` rows when enrichment is complete enough
  - upsert missing article data and `source_article` links
  - publish the final `daily_edition` / `daily_edition_article` state in one transaction
  - keep the old lineup untouched until the final swap when `replace_existing` is true
  - handle the zero-success branches explicitly
- `src/workflows/generate_daily_edition/sandbox/runner.ts`
  - run inside the sandbox
  - fetch the source page, ask Opencode to extract relevant article links, and constrain results by the requested date plus 48-hour lookback
  - visit candidate links and build structured article payloads
  - sign `timestamp + "." + raw_body` with the derived secret and POST to the workflow webhook URL
- `src/lib/server/editions.ts`
  - extend existing helpers so queries cleanly expose the generation states the UI needs, especially `status`, `generated_at`, and article counts
- `src/routes/news/[[date=date]]/+page.svelte`
  - show the generate CTA only when the selected edition is missing or empty
  - wire the CTA to the new remote form
- `src/lib/EditorialDesign.svelte`
  - show a dedicated `generating` state while the workflow is running
  - show explicit empty-edition copy when a published edition has zero articles
- `src/routes/editions/+page.svelte`
  - add a visual treatment for `generating` and `failed` alongside the current `draft` / `published` badge logic
- `src/routes/editions/[date=date]/+page.svelte`
  - surface edition-level generation state and a non-destructive banner if the edition is currently generating
  - do not add rerun controls in v1

## Workflow shape

1. User opens `/news/[date]`.
2. If the edition is missing or empty, the page offers a manual generate action.
3. The remote form authenticates with `get_user()`, validates the date, and starts the workflow through `src/lib/server/edition_generation.ts`.
4. `prepare_generation.ts` creates or updates the `daily_edition` row immediately and marks it `generating`.
5. The workflow loads active sources.
6. For each source, the workflow creates a webhook, launches a sandbox in a step, awaits the webhook request, and verifies/parses the callback in a step.
7. The workflow collects all source results, tolerating individual source failures.
8. `persist_edition.ts` canonicalizes URLs, reuses stored article enrichment where possible, writes missing article data, and publishes the final lineup.
9. The final state is:
   - `published` with articles when at least one source yields usable content
   - unchanged existing edition when a rerun produces zero successes
   - a new empty published edition with explicit no-new-articles copy when no edition existed and zero sources succeed

## Persistence rules

- Reuse only the existing schema: `article`, `source_article`, `daily_edition`, and `daily_edition_article`.
- Treat `article.canonical_url` as the dedupe key across all source results.
- Use `normalize_url` from `src/lib/server/sources.ts` before any article lookup or insert.
- Consider an article row reusable when the stored enrichment is already complete enough for v1 display; fill gaps only when critical fields are missing.
- Do not clear an existing lineup until the final transaction is ready to replace it.
- Keep `replace_existing` in the workflow input even though v1 exposes no rerun UI.

## Security and env strategy

- Keep `WEBHOOK_SECRET` as the root secret in app env only.
- After `createWebhook()` returns a token, derive a scoped secret such as `HMAC_SHA256(WEBHOOK_SECRET, webhook.token)`.
- Pass only that derived secret into the sandbox.
- Require callback headers such as:
  - `x-news-callback-timestamp`
  - `x-news-callback-signature`
- In `consume_source_webhook.ts`, recompute the derived secret from `WEBHOOK_SECRET` and the workflow webhook token, reject stale timestamps, and verify the signature with a timing-safe comparison before accepting the payload.
- Document all required env in `.env.example`, including:
  - `WEBHOOK_SECRET`
  - the Opencode env needed inside the sandbox
  - the Vercel sandbox credentials needed server-side
  - any workflow/runtime origin requirement needed for callback URLs
