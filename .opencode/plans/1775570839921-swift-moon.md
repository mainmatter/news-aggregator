# Per-user article-selection prompt

## Recommended approach

Add a new app-level `user_settings` table with a nullable `article_selection_prompt` field, expose it on the existing `/sources` management page, and thread that value into the generation workflow so it is only used inside `choose_links(...)` in the sandbox selector prompt.

This is the safest fit for the current codebase:

- it keeps app personalization out of the Better Auth `user` table in `src/lib/server/db/auth.schema.ts`
- it avoids custom Better Auth typing work for `get_user()`
- it keeps the change scoped to selection only, which matches the requested behavior

## Files to change

Database and server data access:

- `src/lib/server/db/schema.ts`
- `src/lib/server/settings.ts` (new)
- generated migration under `drizzle/`

Remote functions and UI:

- `src/lib/settings.remote.ts` (new)
- `src/routes/sources/+page.svelte`

Generation pipeline:

- `src/lib/server/edition_generation.ts`
- `src/workflows/generate_daily_edition/types.ts`
- `src/workflows/generate_daily_edition/steps/launch_source_sandbox.ts`
- `src/workflows/generate_daily_edition/sandbox/runner.js`

Verification:

- `e2e/sources.test.ts`
- `src/lib/server/*.spec.ts` if we add unit coverage for new server helpers

## Implementation steps

0. Start with a failing test

- Extend `e2e/sources.test.ts` first so it covers the new prompt behavior on `/sources`.
- The initial test should fail before any feature code exists.
- Cover at least: save, reload persistence, update, and clear.
- If `src/lib/server/settings.ts` ends up with meaningful logic, add a small failing unit test there too before implementing the helper.

1. Add persistent user settings

- Create `user_settings` in `src/lib/server/db/schema.ts` with:
- `user_id` unique FK to `user.id`
- `article_selection_prompt` nullable `text`
- standard timestamps via the existing `create_timestamps()` helper
- Generate and apply the migration with `pnpm db:generate` and `pnpm db:migrate`.

2. Add a small server-side settings module

- Add `src/lib/server/settings.ts` for the app-specific DB access layer.
- Keep it minimal: one read helper and one upsert helper are enough.
- Missing row should behave like `null` prompt, not an error.

3. Add remote query/form for settings

- Add `src/lib/settings.remote.ts` following the existing `query(...)` / `form(...)` pattern used in `src/lib/sources.remote.ts`.
- Expose:
- `get_user_settings`
- `update_user_settings`
- Validation rules:
- trim whitespace
- empty string becomes `null`
- max length `1000`
- reject oversized input with a field error

4. Extend the existing sources management page

- Reuse `src/routes/sources/+page.svelte` instead of adding a new route.
- Add a small section near the top or bottom of the page for the user-level article-selection prompt.
- Keep the UI intentionally narrow in scope:
- one textarea for article-selection guidance
- helper copy that says it affects article choice only, not summary tone/formatting
- save action with persisted reload behavior
- Do not add a new nav item yet; the existing “Manage Sources” entry is enough for this first pass.

5. Thread the setting into generation

- In `src/lib/server/edition_generation.ts`, load the user setting before starting the workflow.
- Extend `EditionGenerationInput` in `src/workflows/generate_daily_edition/types.ts` with optional `article_selection_prompt`.
- Pass that value through the existing workflow input into `launch_source_sandbox.ts`.
- In `launch_source_sandbox.ts`, set a sandbox env var only when the prompt is present, for example `ARTICLE_SELECTION_PROMPT`.

6. Apply it only to link selection

- In `src/workflows/generate_daily_edition/sandbox/runner.js`, read the optional env var.
- Update `choose_links(...)` only.
- Do not change `summarize_article(...)`.
- Inject the user text as a clearly delimited preference block, for example:
- “User article-selection preferences: ...”
- Frame it as relevance guidance, not as an instruction that can change the task or output schema.
- Keep the existing source/date/freshness rules as the authoritative instructions.

## Prompt handling rules

- Treat the field as plain text only.
- Never render it as HTML.
- Trim before save.
- Empty value clears the preference.
- Cap length to control prompt size and abuse.
- In the selector prompt, explicitly say the value is for relevance steering only and must not override the system task, source-verification rules, or output requirements.

## Verification

Run local checks:

- `pnpm check`
- `pnpm test:unit -- --run`

Add focused automated coverage:

- start by writing the `/sources` prompt test so it fails first, then implement until it passes
- extend the existing `/sources` E2E coverage to save, reload, update, and clear the prompt
- unit tests for `src/lib/server/settings.ts` if that module contains non-trivial upsert/defaulting logic

Manual end-to-end verification:

- log in and save a prompt in `/sources`
- reload and confirm the saved value persists
- clear the prompt and confirm it is removed cleanly
- start edition generation and confirm the workflow still succeeds with and without a settings row
- inspect the sandbox input/log path to verify the prompt reaches `choose_links(...)` only

## Risks and notes

- Existing users will have no `user_settings` row, so all reads must default to `null`.
- Putting this field on the Better Auth `user` table would make auth schema regeneration and typing more brittle than necessary.
- User-entered prompt text may appear in logs if we log sandbox payloads verbosely; avoid logging the raw preference unless needed for debugging.
- This change improves per-source article choice, but it does not add a later cross-source reranker. That can stay out of scope for this first pass.
