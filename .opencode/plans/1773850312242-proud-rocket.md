# Edition management plan

## Recommended approach

- Keep `/news/archives` as the archive entry page and add a prominent `Manage Editions` link there.
- Build the authenticated management experience under `/editions`, with:
  - `/editions` for create + list/manage existing editions
  - `/editions/[date]` for editing one edition's metadata and article lineup
- Reuse the remote form pattern from `src/routes/sources/+page.svelte` and `src/lib/sources.remote.ts`: authenticated remotes, `form(...).for(id)`, `FieldErrors`, and optimistic updates where safe.
- Support both adding existing ingested articles and manually creating a new article that is stored as if it had been ingested.

## Files to modify

- `src/routes/news/archives/+page.svelte`
- `src/lib/editions.remote.ts`
- `src/lib/server/editions.ts` (new)
- `src/routes/editions/+page.svelte` (new)
- `src/routes/editions/[date]/+page.svelte` (new)
- `src/lib/EditorialDesign.svelte` (optional extra navigation entry)
- tests for remotes/helpers and e2e coverage

## Implementation steps

1. Replace the mock summary implementation in `src/lib/editions.remote.ts` with authenticated per-user edition queries backed by the database.
2. Add server helpers in `src/lib/server/editions.ts` for:
   - loading owned editions and one owned edition by date/id
   - building an editor DTO that joins `daily_edition`, `daily_edition_article`, `article`, and source-derived display data
   - finding candidate articles from the current user's source universe, excluding articles already in the edition
   - safely reindexing article positions for add/remove/reorder operations
3. Add edition remotes in `src/lib/editions.remote.ts` for:
   - `get_daily_editions()` real summary data
   - `get_manageable_editions()` for `/editions`
   - `get_edition_editor(date)` for `/editions/[date]`
   - `search_editable_articles(...)`
   - `create_manual_article(...)`
   - `create_edition(...)`
   - `update_edition_meta(...)`
   - `add_edition_article(...)`
   - `update_edition_article(...)`
   - `remove_edition_article(...)`
   - `reorder_edition_articles(...)` if drag/reorder is implemented now
4. Update `src/routes/news/archives/+page.svelte` to include a `Manage Editions` CTA while keeping the current archive browsing behavior.
5. Create `src/routes/editions/+page.svelte` using the same shell/style pattern as `src/routes/sources/+page.svelte`:
   - `await get_user()` at the top
   - create-edition form at the top
   - existing editions list below with `Edit` links into `/editions/[date]`
6. Create `src/routes/editions/[date]/+page.svelte` with the same editorial management layout:
   - edition metadata form (`title`, `summary`, `status`, date shown read-only)
   - article search/add section
   - manual article creation form for cases where the story was not already ingested
   - current article cards with inline per-article edit/remove controls
   - optional reorder controls if time allows
7. Keep all mutations authenticated inside the remotes via `await get_user()` and enforce ownership checks server-side before every update/delete.
8. Handle DB uniqueness constraints with friendly validation errors:
   - duplicate edition date for the same user
   - duplicate article in the same edition
   - conflicting positions during reorder/add/remove

## Data shape notes

- Edition summary rows should include: `id`, `edition_date`, `status`, `title`, `summary`, `article_count`, and ideally `updated_at` for the management list.
- The editor payload should include ordered edition articles with both edition-specific override fields and canonical article fields needed for display/search.
- Candidate article search should be scoped to the current user's sources by joining through `source_article` and related source ownership tables.
- Manual article creation should create a real `article` row, attach it to a selected user-owned source via `source_article`, and then add it to the edition so it behaves like ingested content in later searches and edits.
- The manual article form should capture the minimum viable ingestion-like fields: source, canonical URL, title, summary, category, and published date/time if available.

## Verification

- Run targeted tests for new edition server helpers/remotes, especially ownership checks and position reindexing.
- Run the app and verify end-to-end:
  - unauthenticated access to `/editions` and `/editions/[date]` redirects
  - `/news/archives` shows the new `Manage Editions` entry point
  - creating an edition works and duplicate dates show a clear error
  - editing metadata persists correctly
  - adding existing articles, manually creating articles, editing, removing, and (if implemented) reordering articles works without position conflicts
- Run relevant e2e coverage similar to the existing auth/sources flows.

## Notes / risks

- The reader view under `src/routes/news/[[date=date]]/+page.svelte` still uses mock article content today, so edition management may initially update only the management/archive data unless that reader flow is also switched to real edition detail queries.
- Manual article creation needs careful duplicate handling around `article.canonical_url` and should enforce that the chosen source belongs to the current user.
