# Database Design Plan

Recommended approach: model the database in two layers. Keep canonical ingestable content shared across the system (`source`, `article`, source/article queue tables), then add per-user curation tables (`user_source`, `daily_edition`, `daily_edition_article`) so users can share sources and articles without blocking future personalization. Implement the schema with Drizzle's SQL-table style only, using explicit foreign keys and manual SQL joins instead of Drizzle relation helpers.

## Why this approach

- It matches the product direction: sources are reusable, editions are per-user, and historical daily newsletters are first-class.
- It keeps ingestion flexible for AI-based extraction from arbitrary pages, without hard-coding RSS/API/source-type assumptions.
- It avoids duplicating articles for every user while still allowing future user-specific edits through edition-level override fields.
- It supports two separate pipelines from the start: source fetching and article fetching/summarization.
- It fits the current codebase cleanly: Better Auth already owns `user`, and Drizzle schema is centralized in one file.

## Files to change

- `src/lib/server/db/schema.ts`
- `src/lib/schemas.ts`
- `drizzle.config.ts` (verify only; no structural change expected)
- generated migration output from the existing Drizzle workflow

## Recommended schema

1. Add canonical source storage.
   - Create `source` for the real upstream page/feed target.
   - Include `canonical_url`, `display_name`, optional freeform `source_kind`, `last_visited_at`, `last_success_at`, and timestamps.
   - Do not store `is_active` on `source`; active/inactive status should be derived from current subscriptions.
   - Keep fetch timestamps on the shared source because sources are shared across users.

2. Add the user-to-source subscription layer.
   - Create `user_source` as a join table between Better Auth `user` and `source`.
   - Include optional user-specific fields like `label`, `is_active`, and timestamps.
   - Enforce one subscription per user/source pair.
   - Treat `user_source.is_active` as the only persisted activation flag; a source is active when it has at least one active subscription.

3. Add canonical extracted articles.
   - Create `article` as a canonical article record that is not owned by any one source.
   - Store `canonical_url`, `title`, `summary`, freeform `category`, `published_at`, `content_hash`, `summarized_at`, and timestamps.
   - Treat normalized `canonical_url` as the primary dedupe key, with `content_hash` as a fallback for cases where URL canonicalization is imperfect.
   - Keep articles canonical so the same article can appear in multiple users' editions and can be discovered from multiple sources without duplicate processing.

4. Record which sources discovered which articles.
   - Create `source_article` as a join table between `source` and `article`.
   - Use it to preserve provenance when multiple sources point at the same article.
   - Optionally include `first_seen_at` or `discovered_at` so ingestion can track when a source surfaced an article.

5. Add per-user daily newsletter history.
   - Create `daily_edition` linked to `user` with `edition_date`, `status`, optional edition-level `title`/`summary`, `generated_at`, and timestamps.
   - Enforce one edition per user per day so past days are preserved naturally.

6. Add the edition/article curation layer.
   - Create `daily_edition_article` linking `daily_edition` and `article`.
   - Include numeric `position`, optional `section`, `reason`, and override fields like `custom_title`, `custom_summary`, and `custom_category`.
   - Keep `position` as an orderable number so an article can move up or down by updating a single value rather than rewriting a whole list.
   - Use this table to preserve future per-user customization without mutating canonical articles.

7. Add both ingestion queues explicitly.
   - Create `source_fetch_job` for source visits: `source_id`, `status`, `scheduled_for`, `started_at`, `finished_at`, `attempt_count`, `error_message`, and metadata.
   - Create `article_process_job` for fetching/summarizing the discovered article itself: `article_id`, `status`, `scheduled_for`, `started_at`, `finished_at`, `attempt_count`, `error_message`, and metadata.
   - If needed, let `article_process_job` reference the triggering `source_fetch_job` so the pipeline is traceable end-to-end.

## Constraints and indexes

- `source.canonical_url` unique.
- `user_source (user_id, source_id)` unique.
- index `user_source (source_id, is_active)` to efficiently derive whether a source is active.
- `daily_edition (user_id, edition_date)` unique.
- `daily_edition_article (daily_edition_id, article_id)` unique.
- `daily_edition_article (daily_edition_id, position)` unique.
- `article.canonical_url` unique as the primary dedupe key.
- index `article.published_at` and `article.content_hash`.
- `source_article (source_id, article_id)` unique.
- index `source_fetch_job (source_id, status, scheduled_for)`.
- index `article_process_job (article_id, status, scheduled_for)`.
- Use cascading deletes for clearly user-owned rows like `user_source`, `daily_edition`, and `daily_edition_article`.
- Do not use Drizzle `relations(...)`; keep the schema in plain SQL-table style with foreign keys and perform joins in query code.

## Implementation plan

1. Replace the starter `task`-only schema in `src/lib/server/db/schema.ts` with the news domain tables while keeping `export * from './auth.schema';`.
2. Reuse the Better Auth `user` table from `src/lib/server/db/auth.schema.ts` as the owner for `user_source` and `daily_edition`; do not duplicate user profile data.
3. Model article provenance with a `source_article` join table so articles stay canonical and are not tied to a single source.
4. Add both queue tables in the first schema pass so source crawling and article summarization are represented separately from day one.
5. Update `src/lib/schemas.ts` so UI-facing article types no longer assume the current hard-coded `id: number` shape and can represent canonical articles plus edition-specific presentation fields.
6. Keep future query modules under `src/lib/server/db/` join-driven and SQL-oriented, matching your Drizzle preference instead of adding Drizzle relation-based helpers.
7. Generate and apply the schema changes through the existing Drizzle workflow after implementation.

## Verification

- Run `pnpm db:generate` to confirm the schema compiles.
- Run the project migration/push command used in this repo and confirm the new tables appear.
- Run `pnpm check` to catch TypeScript and schema/type mismatches.
- Verify these end-to-end behaviors once query code exists:
  - a user can subscribe to an existing source without duplicating the source row
  - a source becomes active/inactive based on active `user_source` rows rather than a stored `source.is_active`
  - the same article can be linked from multiple sources without duplicating the canonical `article` row
  - source fetching and article summarization can be tracked independently through their own queue tables
  - one user has exactly one edition per day
  - edition-level overrides do not mutate the canonical `article` row
