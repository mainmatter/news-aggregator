# Per-user article-selection prompt

## Recommended approach

Add a new app-level per-user settings row with a nullable `article_selection_prompt`, expose it on `/sources` as a dedicated saved textarea above the source list, and use it only in `choose_links(...)` alongside the existing per-source `label` field.

Key decisions locked in:

- one global saved user setting, not a per-run override
- place the new textarea section above the source list
- explicit Save button
- trim leading/trailing whitespace, preserve internal line breaks
- max length `1000`
- empty value clears the prompt
- never log the raw prompt
- source `label` also influences selection
- combine as global guidance plus source-specific hint
- keep existing source label UI copy unchanged
- helper copy for the new textarea mentions only the global prompt
- labels remain soft topical hints, not hard filters
- apply both prompt and labels only in `choose_links(...)`, not `summarize_article(...)`
- existing saved labels take effect immediately after rollout
- read the global prompt inside the workflow, matching current source-loading behavior

## Implementation steps

1. Start with a failing `/sources` test

- Extend `e2e/sources.test.ts` first.
- Cover the new global prompt form on `/sources`:
- save
- reload persistence
- update
- clear
- trimming of leading/trailing whitespace
- preservation of internal line breaks

2. Add persistent user settings

- Create `user_settings` in `src/lib/server/db/schema.ts` with:
- `id` via `create_id()`
- `user_id` unique FK to `user.id` with `onDelete: 'cascade'`
- `article_selection_prompt` nullable `text`
- timestamps via `create_timestamps()`
- Generate and apply the migration with `pnpm db:generate` and `pnpm db:migrate`.
- Missing row must behave like `null`, not an error.

3. Add a minimal server settings module

- Add `src/lib/server/settings.ts`.
- Keep it small:
- one read helper returning `{ article_selection_prompt: string | null }`
- one upsert helper keyed by `user_id`
- Normalize prompt input there or immediately before calling it:
- trim edges
- preserve internal newlines
- empty string becomes `null`

4. Add remote query/form for the user setting

- Add `src/lib/settings.remote.ts` following the existing `query(...)` / `form(...)` pattern.
- Expose:
- `get_user_settings`
- `update_user_settings`
- Validation rules:
- optional string
- trim whitespace
- preserve internal line breaks
- empty string becomes `null`
- max length `1000`
- reject oversized input with a field error

5. Extend `/sources`

- Load `get_user_settings()` alongside `get_user_sources()`.
- Add a dedicated section above the source list.
- Keep the UI narrow:
- one textarea for global article-selection guidance
- helper copy that says it affects article choice only, not summary tone/formatting
- explicit Save button
- empty textarea + Save clears the stored value
- persisted reload behavior
- Do not add a new nav item.
- Keep the existing per-source `Label (optional)` field copy unchanged.

6. Read settings inside the workflow

- Add `src/workflows/generate_daily_edition/steps/get_user_generation_settings.ts`.
- In `src/workflows/generate_daily_edition/index.ts`, load settings near `get_user_sources(input.user_id)`.
- Do not widen the top-level workflow launch payload just to carry the prompt.
- Keep `WorkflowUserSource.label` as-is; it already exists and will now be used downstream.

7. Pass prompt and label into the sandbox

- In `launch_source_sandbox.ts`, add optional env vars only when values are present:
- `ARTICLE_SELECTION_PROMPT`
- `SOURCE_LABEL`
- Do not log either value.

8. Apply both only in `choose_links(...)`

- In `src/workflows/generate_daily_edition/sandbox/runner.js`, read the optional env vars.
- Update `choose_links(...)` only.
- Do not change `summarize_article(...)`.
- Keep the current source/date/freshness/direct-link rules authoritative.
- Inject both values as clearly delimited advisory context, for example:
- `User guidance for article selection: ...`
- `Source label: ...`
- Add explicit guardrail text that both are soft relevance hints and must not override task requirements, source verification, freshness rules, or output shape.

## Prompt handling rules

- Treat the saved prompt as plain text only.
- Never render it as HTML.
- Never log the raw saved prompt.
- Preserve internal line breaks after trimming.
- Empty value clears the global setting.
- Use both the global prompt and source label as soft prioritization hints only.
- Do not treat either as a hard filter.
- Existing saved source labels should begin influencing selection immediately after rollout.

## Verification

Run local checks:

- `pnpm check`
- `pnpm test:unit -- --run`
- `pnpm test:e2e`

Focused automated coverage:

- extend `e2e/sources.test.ts` for global prompt save/reload/update/clear

Manual verification:

- log in and save a global prompt in `/sources`
- reload and confirm the saved value persists
- clear the prompt and confirm it is removed cleanly
- edit a source label and confirm the saved value still persists normally
- start edition generation with and without a saved global prompt
- confirm generation still succeeds when no `user_settings` row exists
- inspect the sandbox execution path and confirm only `choose_links(...)` receives the global prompt and source label
- confirm no raw prompt text is emitted to logs

## Risks and notes

- Existing users will often have no `user_settings` row, so reads must default to `null`.
- Existing source labels will begin affecting article selection immediately, which is a real behavior change.
- Because source label copy stays unchanged, that behavior change will be only partially discoverable from the UI.
- Uncapped labels can still be noisy input even if they are intended as short tags.
- Streaming sandbox stdout/stderr means any future debug logging in `runner.js` could leak prompt text unless avoided deliberately.
- This remains a selection-only feature; it does not personalize summarization or add a later cross-source reranker.
