# Sources management plan

## Recommended approach

Build `/sources` as an authenticated SvelteKit remote-functions page, following the same pattern as the existing auth/news routes: the page itself calls `await get_user()` and a `query(...)`, while all mutations go through `form(...)` remote functions in a new `src/lib/sources.remote.ts` module.

The page should manage each signed-in user's saved sources through `user_source`, while reusing shared `source` rows by normalized `canonical_url` under the hood.

## Files to add or change

- `src/lib/sources.remote.ts`
  - Add the authenticated remote query/forms for listing, creating, updating, and deleting user sources.
  - Keep the Valibot schemas here unless client preflight validation becomes necessary.
- `src/lib/server/sources.ts`
  - Add small server helpers for URL normalization, source lookup/create, ownership checks, and optional orphan cleanup.
- `src/routes/sources/+page.svelte`
  - Add the authenticated page UI, add form, current sources list, inline edit state, and delete controls.
- `src/lib/server/db/schema.ts`
  - Read-only reference for the existing tables; no schema change is planned right now.

## Remote functions

- `get_user_sources = query(async () => ...)`
  - Call `await get_user()`.
  - Query `user_source` joined with `source`, filtered by the current `user.id`.
  - Return UI-ready rows such as `user_source_id`, `source_id`, `canonical_url`, `display_name`, `source_kind`, `label`, `is_active`, `created_at`, `updated_at`.
- `create_user_source = form(schema, async (data, issue) => ...)`
  - Call `await get_user()`.
  - Normalize `canonical_url`.
  - Find or create the shared `source` row.
  - Insert the user's `user_source` row, or reactivate/update it if the pair already exists.
  - Refresh `get_user_sources()` server-side so the page gets the updated list in one round trip.
- `update_user_source = form(schema, async (data, issue) => ...)`
  - Call `await get_user()`.
  - Verify the submitted `user_source` row belongs to the current user.
  - Update per-user fields (`label`, `is_active`).
  - If the URL changes, normalize it, find/create the target `source`, and relink `user_source.source_id` instead of mutating the existing shared `source` in place.
  - Refresh `get_user_sources()`.
- `delete_user_source = form(schema, async (data) => ...)`
  - Call `await get_user()`.
  - Verify ownership.
  - Delete only the current user's `user_source` row.
  - Optionally remove the `source` row afterward if it is orphaned.
  - Refresh `get_user_sources()`.

## Data-model rules

- Treat `source` as shared source identity/metadata.
- Treat `user_source` as the user's saved subscription/config row.
- For safety on a per-user page, editing should primarily affect `user_source`.
- Recommended editable fields:
  - `canonical_url` (implemented as relink-to-source, not in-place mutation)
  - `label`
  - `is_active`
- Recommended create-only or read-only shared fields for now:
  - `display_name`
  - `source_kind`
- This avoids one user silently changing shared metadata for every other user.

## Page implementation in `src/routes/sources/+page.svelte`

- No need to guard the route with `await get_user()` since it is called inside `get_user_sources`.
- Load the current list with `const user_sources = await get_user_sources()`.
- Reuse the existing editorial page language/components where it fits:
  - `Masthead`
  - `SectionRule`
  - `PageFooter`
  - `FieldErrors`
- Add an "Add source" form at the top.
- Render existing sources below as rows/cards with:
  - source title + URL
  - per-user label/active status
  - edit and delete actions
  - editing should be available even without JS so no client side magic to switch to an edit form
- Use isolated form instances with `.for(row.user_source_id)` for per-row edit/delete state.
- Prefer `form.enhance(async ({ form, submit }) => { await submit().updates(get_user_sources()); ... })` so add/edit/delete update only the sources query and avoid full-page reloads when JS is available.
- Keep native form fallback intact for no-JS submissions.
- When styling the page, follow the existing `layout.css` tokens and shared editorial spacing/typography.
- Use optimistic updates for the add/edit and remove form.

## Validation and error handling

- Validate `canonical_url` as required, trimmed, and URL-shaped.
- Normalize URLs consistently before lookups/inserts so the unique constraint on `source.canonical_url` works for deduping.
- Treat `label` as optional and trimmed.
- Treat `is_active` as an optional boolean with a default of `true`.
- On update/delete, look up by `user_source.id` plus current `user.id`; never trust the client-submitted id alone.
- Convert uniqueness collisions into friendly form errors instead of generic failures.
- Surface field issues through `FieldErrors` and any form-level error near the relevant form.

## Verification

- Manual checks
  - unauthenticated `/sources` redirects to `/`
  - authenticated `/sources` shows the page shell and current saved sources
  - adding a new URL creates or reuses `source` and creates/reactivates `user_source`
  - editing label/active state persists immediately
  - editing the URL relinks the saved source correctly
  - deleting a source removes only the current user's saved row
- Code checks
  - run `pnpm run check`
  - run targeted unit/component tests if added for source helpers or page behavior
  - run a focused Playwright flow for auth + `/sources` CRUD if the implementation adds e2e coverage
  - consider a TDD approach of writing a failing test before the implementation to verify that the test actually verify something
