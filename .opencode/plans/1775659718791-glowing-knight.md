# Docker/Vercel Sandbox Backend Abstraction

## Goal

Add a small sandbox abstraction around the current runtime lifecycle so `SANDBOX_BACKEND=vercel|docker` can switch between the existing `@vercel/sandbox` path and a Docker-backed path for local development and CI.

The Docker path should preserve the current async contract: one isolated runtime per source, `runner.js` posts the signed webhook callback, and the workflow still waits on `createWebhook()` + timeout.

## Recommended approach

1. Extract the current Vercel-specific logic behind a backend interface that mirrors the subset of the Vercel sandbox API we actually use, especially `runCommand`, so command execution is orchestrated in shared code rather than hard-coded inside each backend.
2. Centralize the shared lifecycle currently embedded in `launch_source_sandbox.ts`:
   - sandbox creation config
   - `writeFiles([...])`
   - `runCommand('npm', ['i', ...])`
   - `runCommand('npm', ['i', '-g', ...])`
   - detached `runCommand({ cmd: 'node', args: ['runner.js'], ... })`
   - stop/wait flow used by `stop_sandbox.ts`
3. Centralize the shared helper logic currently embedded in `launch_source_sandbox.ts`:
   - webhook URL rewriting with `tunnel_base_url`
   - callback secret derivation from `WEBHOOK_SECRET`
   - edition window calculation
   - sandbox/container env assembly
   - in-runtime file payload assembly (`package.json`, `runner.js`)
4. Keep the workflow orchestration in `src/workflows/generate_daily_edition/index.ts` almost unchanged, but have the step layer work through the shared sandbox interface so both backends share the same command sequence and cleanup path.
5. Keep Vercel as the default backend initially to avoid changing production behavior.

Resolved decisions from review:

- backend selection is strict:
  - unset/empty `SANDBOX_BACKEND` => `vercel`
  - `vercel` or `docker` => use exactly that backend
  - any other value => throw a configuration error
  - if `docker` is selected and Docker is unavailable or image setup fails, surface a hard launch error with no fallback
- keep the abstraction small and behavior-focused:
  - separate backend, sandbox, and command handle types
  - `runCommand` uses a single object-based API
  - `writeFiles` accepts `string | Uint8Array`
  - no first-class log retrieval API in v1
  - keep working directory out of the public interface in v1
- keep orchestration in `launch_source_sandbox.ts`; only hide backend selection/implementation behind a separate module
- shared helpers should own callback secret derivation so launch and webhook consumption use the same logic
- update `consume_source_webhook.ts` to import the shared callback-secret helper rather than duplicating it
- create config should be backend-agnostic and include `timeout_ms`; Vercel may use it while Docker ignores it in v1
- `stop()` must be idempotent and tolerate missing runtimes during cleanup; `get()` can stay strict for normal use
- keep backend resolution at call time in v1; changing `SANDBOX_BACKEND` while workflows are already in flight is unsupported and may break cleanup
- Docker support is v1-local-development-and-CI only; production behavior remains Vercel-first
- keep Docker containers fully ephemeral in v1; do not add host mounts or shared writable caches
- reject `sudo` in the Docker backend in v1 unless a concrete need appears
- accept limited detached-command log replay in v1; lifecycle/exit semantics matter more than durable stdout/stderr retrieval
- preserve current per-source parallelism in workflow orchestration
- pin the default Docker image to a specific `node:24-bookworm-slim@sha256:...` digest in code and mirror that pinned value in `.env.example`
- keep cleanup ordering conceptually the same (`getCommand(id).wait()` before `stop()`), with no extra cleanup deadline in v1; treat the unified timeout as best-effort rather than a hard wall-clock bound
- add an opt-in Docker-only debug escape hatch such as `DOCKER_SANDBOX_KEEP_FAILED=1` that preserves failed containers only on failure paths

## Backend shape

Create a new sandbox module group under `src/workflows/generate_daily_edition/sandbox/`:

- `types.ts`
  Defines the backend interface and the sandbox/command handles. Keep the interface minimal and behavior-oriented, for example:
  - backend object includes `name: 'vercel' | 'docker'`
  - `create(config)`
  - `get(id)`
  - `writeFiles(files)`
  - `runCommand({ ... })`
  - `getCommand(id)`
  - `stop(...)`
- `shared.ts`
  Holds shared helpers and constants extracted from the current launch step.
- `backend.ts`
  Reads `env.SANDBOX_BACKEND` at call time and returns the selected backend implementation.
- `vercel_backend.ts`
  Thin adapter over `@vercel/sandbox`.
- `docker_backend.ts`
  Implements the same interface on top of Docker using a Node Docker client.
- `runner_files.ts`
  Returns the generated `package.json` + raw `runner.js` payload used by both backends.

- Keep the shared command sequence in `launch_source_sandbox.ts` for now. Do not extract `orchestrate.ts` unless implementation reveals a clear need.

Then reduce these step files to thin delegates around the shared sandbox interface:

- `src/workflows/generate_daily_edition/steps/launch_source_sandbox.ts`
- `src/workflows/generate_daily_edition/steps/stop_sandbox.ts`

## Docker implementation

Use `dockerode` with `@types/dockerode`. If ESM interop is awkward, keep the smallest workaround contained entirely inside `docker_backend.ts`.

Map Docker to the current semantics like this:

1. Create one ephemeral container per source run.
2. Start from an official Node image (Node 24 to match Vercel runtime) with no checked-in `Dockerfile`.
   - default image: a pinned `node:24-bookworm-slim@sha256:...` digest
   - explicitly ensure the image exists and pull it if missing before creating the container
3. Programmatically upload `package.json` and `runner.js` into the container filesystem through the shared `writeFiles` API.
   - use Docker archive upload (`putArchive`) into a fixed internal workspace such as `/workspace`
   - reject absolute paths and `..` traversal in `writeFiles` paths
4. Implement `runCommand` so the shared orchestration can invoke the same install and execution commands it currently invokes against Vercel.

- treat `cmd` and `args` as trusted internal inputs; no extra validation is required in v1
- ignore only best-effort observability options like `stdout`/`stderr` if Docker cannot support them cleanly; throw for unsupported behavioral options
- reject `sudo` explicitly in v1 rather than pretending it maps cleanly to Docker privileges

5.  Keep the container alive with a simple idle main process such as `sleep infinity`, then map command execution onto `docker exec`.

- the detached runner command should preserve the existing behavior contract and may use a synthetic command ID if needed, as long as `getCommand(id).wait()` semantics still work after reloading from `sandbox_id`
- follow Vercel semantics for `wait()`: return a finished command/result with `exitCode` rather than throwing on non-zero exit

6. Return the container ID as `sandbox_id`.
7. In `stop`, inspect/stop/remove the container; missing or already-exited containers should be non-fatal.
8. Always remove the container after cleanup; do not keep failed containers around by default.
   - optional debug escape hatch: if `DOCKER_SANDBOX_KEEP_FAILED=1`, preserve failed containers on failure paths only

Important details:

- keep the callback URL flow unchanged; the Docker container should still call back to the workflow webhook using the same signed payload logic already implemented in `runner.js`
- keep the existing tunnel/default callback path as the only v1 callback-routing strategy; `DOCKER_SANDBOX_NETWORK` only controls network attachment and does not introduce a separate callback-host override in v1
- avoid burying `npm install` / `node runner.js` inside the Docker backend; those should stay in shared orchestration through the backend's `runCommand` API
- replace the current 1-minute/15-minute mismatch with one shared 15-minute timeout value owned by workflow/orchestration and mirrored into backend create config; because cleanup remains unbounded in v1, treat this as a best-effort deadline rather than a hard upper bound
- if launch fails after partially creating a runtime/container, perform best-effort cleanup before rethrowing

## Env/config changes

Update env handling and docs to include:

- `SANDBOX_BACKEND=vercel|docker`
- `.env.example` should show `SANDBOX_BACKEND="docker"` for local discoverability even though the runtime default remains Vercel when unset
- `DOCKER_SANDBOX_IMAGE` with a pinned default `node:24-bookworm-slim@sha256:...`
- optional Docker connection/network overrides only if needed during implementation:
  - `DOCKER_HOST` (documented, but rely on the Docker client/library's normal environment-based resolution rather than custom parsing unless implementation proves otherwise)
  - `DOCKER_SANDBOX_NETWORK`
- `DOCKER_SANDBOX_KEEP_FAILED=1` as an optional Docker-only debug flag

Network behavior decisions:

- keep the existing tunnel-based callback URL rewriting as the default local-dev path
- support `DOCKER_SANDBOX_NETWORK` as an optional override; if set, attach the container to that network
- do not add a first-class callback host override in v1

Keep existing Vercel env vars untouched:

- `VERCEL_TOKEN`
- `VERCEL_PROJECT_ID`
- `VERCEL_TEAM_ID`

Compatibility note:

- preserve the current silent partial-ignore behavior for Vercel credentials; do not tighten this validation in this change
- treat `DOCKER_HOST` support as documented library pass-through only in v1; do not add special parsing or an expanded validation matrix

Keep existing Opencode env passthrough untouched.

## Risks to watch during implementation

- Docker startup/install time may be slower than Vercel sandbox startup.
- Callback reachability from Docker may differ from Vercel in local setups.
- Container cleanup must be idempotent to avoid leaks on timeout/cancellation.
- Shared file/env assembly must stay single-sourced so Docker and Vercel do not drift.
- Per-source parallelism may create noticeable local CPU/memory pressure in Docker mode.
- Because cleanup still waits before stopping with no added deadline, actual wall-clock completion may exceed the nominal timeout budget.
