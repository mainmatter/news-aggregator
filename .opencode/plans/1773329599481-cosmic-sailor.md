# Isolated E2E Database Plan

Recommended approach: use a dedicated `.env.test` pointing at a fixed test-only SQLite/libSQL `file:` database, bootstrap that database once before Playwright starts, then reset it around every test through a custom Playwright fixture that also exposes the DB instance to tests. This keeps the tests truly end-to-end, avoids touching the dev DB, removes cross-run leakage, and makes each test independent. It is inspired by the blog post, but avoids one unnecessary part of it: the app should keep using SvelteKit env access, while tests use a DB factory/helper that does not depend on `$env/dynamic/private`.

## Why this approach

- The blog post is directionally right: Playwright fixtures are the right place for DB setup/cleanup, and a test-only DB is much better than sharing `local.db`.
- A per-run temp DB alone is not enough, because tests would still share state within the suite.
- A per-test DB or request-scoped DB would be more invasive here because the app currently boots one preview server with one singleton DB.
- The clean compromise is: one fixed test DB from `.env.test` + automatic reset/seed around each test + `workers: 1` for determinism.

## Files to change

- `playwright.config.ts`
- `package.json`
- `src/lib/server/db/index.ts`
- `e2e/auth.test.ts`
- `e2e/demo.test.ts`
- new helper such as `src/lib/server/db/create_db.ts`
- new Playwright fixture module such as `e2e/test.ts`
- new env file `/.env.test`

## Implementation plan

1. Extract DB creation into a shared helper.
   - Add a small factory that accepts `url` and `auth_token` and returns the Drizzle/libSQL client.
   - Keep `src/lib/server/db/index.ts` as the app singleton, still reading from `$env/dynamic/private` and calling that factory.
   - This lets Playwright-side code create its own DB client without importing SvelteKit-only env modules.

2. Start Playwright against a dedicated test DB from `.env.test`.
   - Add `.env.test` with a test-only `DATABASE_URL`, `DATABASE_AUTH_TOKEN`, and an E2E-safe `ORIGIN`.
   - Update `package.json` so `test:e2e` loads `.env.test`, runs `drizzle-kit push` first to create/update the DB file, then starts `playwright test`.
   - Because the fixture will wipe the DB around each test, reusing the same `test.db` file across runs is fine.

3. Add a custom Playwright fixture that resets DB state per test and exposes the DB.
   - Create `e2e/test.ts` that extends Playwright `test`.
   - Build a DB client from `process.env.DATABASE_URL` using the shared factory.
   - Expose `db` (and likely `schema`) from the fixture so tests can insert/delete records before navigation when that is the clearest setup.
   - Add an auto fixture that clears all tables before and after each test so each test starts from a known empty state; optionally support explicit seed data for future tests.
   - Prefer deterministic explicit inserts over random seed data for auth tests.

4. Serialize the E2E suite and fix Playwright config gaps.
   - Set `workers: 1` because one preview server + one mutable DB cannot safely support parallel mutation with fixture-driven resets.
   - Add `use.baseURL` and make host/port/origin consistent with preview.
   - Keep `webServer`, but ensure it inherits the `.env.test` values loaded by the `test:e2e` command.

5. Make the auth tests independent.
   - Switch E2E specs to import the custom `test` fixture module.
   - Remove the shared run-wide user/email pattern from `e2e/auth.test.ts`.
   - Each test should either register its own user or use per-test seed data from the fixture.
   - Update any other E2E tests so they also go through the auto-reset fixture.

## Verification

- Run `pnpm test:e2e` and confirm all E2E tests pass.
- Run `pnpm test:e2e` a second time immediately and confirm it still passes without unique-email hacks.
- Confirm the auth suite can run tests individually and in a different order without depending on previously-created users.
- Confirm Playwright uses the expected base URL and that auth redirects still work under the E2E `ORIGIN`.
